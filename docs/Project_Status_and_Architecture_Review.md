## 2. Architecture Assessment: Real vs. Simulated

A critical requirement of the ContextGuard build is to avoid "dummy code" and ensure the platform operates as a legitimate enterprise security tool. Below is the architectural breakdown of the system's fidelity.

### 🟢 100% Real-World Components (Production Grade)
*   **The DPI Layer (Lobster Trap):** This is the actual compiled Veea proxy binary. It is actively intercepting HTTP traffic, parsing real YAML policies, evaluating regexes in real-time, and physically dropping traffic.
*   **The Gemini Risk Engine:** The backend makes real API network calls to `gemini-2.0-flash`. Threat classifications and alert messages stored in the database are dynamically generated by the LLM based on actual intercepted payloads.
*   **The Webhook Bridge & Database:** The bridge utilizes real Python file tailing and real SHA-256 cryptographic hashing to redact prompts before storing structured telemetry in a real SQLite relational database.

### 🟡 Robust Production Fallbacks (High Availability)
*   **Gemini Rate-Limit Degradation:** If the Gemini API crashes or rate-limits the backend (15 req/min free tier limit), the backend catches the error and falls back to using the default severity score provided by the Lobster Trap proxy. This is not "dummy code"; it is a standard enterprise **Fail-Open / Graceful Degradation** pattern ensuring the security pipeline never drops an event during upstream outages.

### 🟠 Simulated Components (By Design / SRS Constraints)
*   **AI Agent Traffic (`simulate_traffic.py`):** Because we do not have a live enterprise AI agent (e.g., a custom Slackbot) running constantly, we use a Python script to send synthetic HTTP requests through the proxy to trigger the policies.
*   **The OAuth Scanner (F1):** According to **SRS Section 8.1 Constraints**, because we do not possess enterprise Google Workspace Admin API tokens, the SRS explicitly mandates: *"Pre-load demo with 10 realistic synthetic OAuth app records."* Therefore, the initial OAuth app list will be synthetic, but the Gemini risk scoring applied to those apps will be dynamically generated.

---

### Summary Conclusion
The core infrastructure, proxy layer, AI analysis, and backend routing are **100% real and production-ready**. The only "simulated" aspects of ContextGuard are the *inputs* into the system (synthetic traffic generation and synthetic initial OAuth app names), which strictly adheres to the defined hackathon constraints.
